# 標準テンプレートの用紙横拡張 設計メモ (別タスク)

Status: メモのみ (未着手)。columns超過対応 (column_overflow_design.md) とは別問題。

## 背景・要望

標準A3テンプレートで初期カラム数より列が多い場合、現状は列幅を圧縮して
収めている。これを「カラム幅は維持し、用紙自体を横に拡張する」挙動にしたい。

- 列圧縮は読みにくい
- 追加列は追加スペースとして扱いたい
- 外部テンプレの columns超過 (pageChunks/gengaDouga) とは別の問題
  (あちらは固定サイズのテンプレ画像にページ分割で対応。こちらは標準A3の
   用紙そのものを可変幅にする話)

## 期待挙動

- ACTION/CELL/SOUND/CAMERA いずれかの列数が初期想定を超えたら、
  その分だけ用紙(描画キャンバス/Preview/書き出し)を横に広げる。
- 既存の列は幅を変えない。

## 影響範囲 (要検討)

広範囲に影響するため、着手前に各点の設計が必要:

1. **Edit画面キャンバスサイズ** — baseWidth 等が固定前提。可変化が必要。
   sections の x 座標計算、ズーム/フィット、横スクロール領域。
2. **Preview 標準A3 用紙サイズ** — TEMPLATE.WIDTH_MM が 297 固定。
   列数に応じて WIDTH_MM を動的算出 (mm/列 × 追加列分を加算)。
   A3 比率が崩れる (横長 A3+ になる) ことの許容可否。
3. **PNG/JPG/PSD 書き出しサイズ** — renderTemplate の canvas 幅、DPI 換算。
   PSD のレイヤー座標も連動。
4. **手書きレイヤー座標** — 手書きは用紙座標系に紐づく。用紙幅変更時の
   既存手書きの座標維持/再マッピング。
5. **スクロール領域** — Edit/Preview とも横スクロール対応。
6. **TDTS/XDTS 出力** — これらは列数を持つデータ形式。用紙幅は出力に
   直接は影響しないはずだが、ラウンドトリップ確認が必要。
7. **印刷/画像出力が横長になることの許容** — A3 印刷前提が崩れる。
   ユーザーへの提示 (用紙が A3 に収まらない旨) が要るか。
8. **どのセクションの列増加で横拡張するか** — ACTION/CELL/SOUND/CAMERA の
   どれを対象にするか。全部か、ACTION/CELL のみか。

## 進め方

- column_overflow Phase C-5 完了後、または独立した別 PR で設計から着手する。
- まず「Preview 標準A3 の用紙幅可変化」だけを最小スコープで試作し、
  Edit/書き出し/手書きへ段階展開するのが安全。
